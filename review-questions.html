C# / .NET Interview Questions

Zagadnienia z bazy pytań. Każde pytanie można rozwinąć: najpierw masz krótką odpowiedź, potem bardziej rozbudowane wyjaśnienie.

C# language & runtime

Difference between out, in and ref
Krótko

ref przekazuje zmienną przez referencję do odczytu i zapisu, out służy głównie do zwracania wartości z metody, a in przekazuje argument przez referencję tylko do odczytu.

Szerzej

ref wymaga, żeby zmienna była zainicjalizowana przed wywołaniem i pozwala metodzie ją zmienić. out nie wymaga wcześniejszej inicjalizacji, ale metoda musi przypisać wartość przed zakończeniem. in jest przydatne szczególnie przy dużych strukturach, bo unika kopiowania, ale chroni argument przed modyfikacją.

void Change(ref int x) => x++;
void Create(out int x) => x = 10;
void Read(in BigStruct value) { /* read only */ }
Using statement
Krótko

using zapewnia automatyczne wywołanie Dispose() na obiekcie implementującym IDisposable.

Szerzej

Używa się go do zasobów, które trzeba jawnie zwolnić, np. plików, połączeń bazodanowych, streamów albo obiektów trzymających uchwyty systemowe. Nawet jeśli wystąpi wyjątek, Dispose() zostanie wywołane, bo kompilator tłumaczy to na blok try/finally.

using var stream = File.OpenRead("file.txt");
// stream.Dispose() wykona się automatycznie na końcu zakresu
Using statement and IL
Krótko

using kompiluje się do konstrukcji podobnej do try/finally, gdzie w finally wywoływany jest Dispose().

Szerzej

To nie jest magiczny mechanizm runtime, tylko składnia ułatwiająca bezpieczne zwalnianie zasobów. Dla klasycznego using kompilator generuje kod, który najpierw tworzy obiekt, potem wykonuje blok, a na końcu niezależnie od wyniku sprawdza, czy obiekt nie jest null i wywołuje Dispose().

var resource = new Resource();
try
{
    resource.Work();
}
finally
{
    resource?.Dispose();
}
Why use StringBuilder in C#?
Krótko

StringBuilder jest używany, gdy wielokrotnie modyfikujemy tekst, bo string jest niemutowalny i każda zmiana tworzy nowy obiekt.

Szerzej

Przy kilku konkatenacjach zwykły + jest OK. Problem pojawia się w pętlach lub przy budowaniu dużego tekstu, bo wiele operacji na string generuje dużo tymczasowych obiektów i zwiększa presję na GC. StringBuilder trzyma bufor, który można modyfikować efektywniej.

var sb = new StringBuilder();
foreach (var item in items)
    sb.AppendLine(item.Name);

string result = sb.ToString();
Async void
Krótko

async void jest złe w większości przypadków, bo nie da się go awaitować i trudniej obsłużyć wyjątki. Wyjątek: event handlery.

Szerzej

Metody asynchroniczne powinny zwykle zwracać Task albo Task<T>. Dzięki temu kod wywołujący może poczekać na zakończenie, obsłużyć wyjątek i testować metodę. async void jest akceptowalne głównie w UI/eventach, gdzie sygnatura wymaga void.

// Dobrze
public async Task SaveAsync() { await repository.SaveAsync(); }

// Tylko dla event handlerów
private async void Button_Click(object sender, EventArgs e) { await SaveAsync(); }
Static constructor
Krótko

Statyczny konstruktor inicjalizuje statyczny stan typu i uruchamia się automatycznie raz, przed pierwszym użyciem typu.

Szerzej

Nie ma parametrów, nie można go wywołać ręcznie i nie może mieć modyfikatora dostępu. Jest używany np. do przygotowania statycznych pól, cache albo konfiguracji typu. Trzeba uważać, bo wyjątek w statycznym konstruktorze może spowodować TypeInitializationException i typ może być nieużywalny w danym AppDomain/procesie.

class Settings
{
    public static readonly string AppName;

    static Settings()
    {
        AppName = "MyApp";
    }
}
Struct vs Class
Krótko

struct to value type, zwykle kopiowany przez wartość. class to reference type, gdzie zmienna przechowuje referencję do obiektu.

Szerzej

Struktury są dobre dla małych, prostych, często niemutowalnych wartości, np. DateTime, Guid, małe value objecty. Klasy są lepsze dla obiektów z tożsamością, większym stanem, dziedziczeniem i cyklem życia. Duże mutowalne struktury mogą powodować błędy i kosztowne kopiowanie.

  • struct: value type, nie może dziedziczyć po innej strukturze lub klasie, domyślnie nie-nullowalny.
  • class: reference type, obsługuje dziedziczenie, może być null.
C# Heap vs Stack
Krótko

Stack przechowuje m.in. ramki wywołań i lokalne wartości, a heap przechowuje obiekty zarządzane przez GC.

Szerzej

Stack jest szybki i działa według kolejności wywołań metod. Heap jest używany dla obiektów, które mogą żyć dłużej niż pojedyncza metoda. Ważne: stwierdzenie "value type zawsze na stacku" jest uproszczeniem. Value type może być częścią obiektu na heapie, np. jako pole klasy albo po boxingu.

Func<>, Action<> and Predicate<>
Krótko

To wbudowane delegaty: Func zwraca wartość, Action nic nie zwraca, a Predicate<T> zwraca bool.

Szerzej

Delegaty pozwalają przekazywać metody jako argumenty. Func<T, TResult> często pojawia się w LINQ, Action<T> przy operacjach typu "wykonaj coś", a Predicate<T> przy filtrowaniu warunkowym.

Func<int, int> square = x => x * x;
Action<string> log = msg => Console.WriteLine(msg);
Predicate<int> isEven = x => x % 2 == 0;
What is "Friend Assembly"?
Krótko

Friend Assembly pozwala jednej assembly widzieć elementy internal z drugiej assembly przez atrybut InternalsVisibleTo.

Szerzej

Najczęściej używa się tego w testach, gdy projekt testowy ma testować klasy lub metody oznaczone jako internal, bez robienia ich publicznymi. Atrybut dodaje się zwykle w projekcie produkcyjnym.

[assembly: InternalsVisibleTo("MyProject.Tests")]
What is default System.String?
Krótko

Domyślna wartość string to null, bo System.String jest typem referencyjnym.

Szerzej

string jest aliasem dla System.String. Mimo że zachowuje się trochę jak wartość, np. jest niemutowalny i ma przeciążony operator ==, nadal jest reference type. Dlatego default(string) daje null, a nie pusty tekst.

string s = default; // null
string empty = string.Empty; // ""
throw ex vs throw
Krótko

throw zachowuje oryginalny stack trace, a throw ex go resetuje i utrudnia debugowanie.

Szerzej

W bloku catch, jeśli chcesz przekazać wyjątek dalej, używaj samego throw;. throw ex; traktuje wyjątek jak rzucony od nowa z aktualnego miejsca, przez co można stracić informację, gdzie błąd powstał pierwotnie.

catch (Exception ex)
{
    logger.LogError(ex, "Error");
    throw; // dobrze
}
Span<T> vs Memory<T>
Krótko

Span<T> jest szybkim widokiem na pamięć, ale działa tylko na stacku. Memory<T> może być przechowywane na heapie i używane z async.

Szerzej

Span<T> to ref struct, więc nie może być polem klasy, nie może być boxowany i nie może przechodzić przez await. Dzięki temu jest bardzo bezpieczny i wydajny do pracy z fragmentami tablic, stringów lub pamięci. Memory<T> jest mniej ograniczone i nadaje się do API asynchronicznych, a gdy potrzebujesz bezpośredniego dostępu, możesz wywołać .Span.

What is Reflection and when to use it?
Krótko

Reflection pozwala odczytywać metadane typów i dynamicznie tworzyć lub wywoływać elementy w runtime.

Szerzej

Możesz sprawdzić właściwości, metody, atrybuty, konstruktory albo dynamicznie utworzyć obiekt. Używa się tego w serializerach, ORM-ach, DI containerach, frameworkach testowych i pluginach. Minusy: jest wolniejsze, mniej bezpieczne typowo i trudniejsze w utrzymaniu niż zwykły kod.

var type = typeof(User);
var props = type.GetProperties();

Memory, GC & resources

Strong vs weak reference in GC
Krótko

Strong reference utrzymuje obiekt przy życiu, a weak reference pozwala GC usunąć obiekt, jeśli nie ma do niego silnych referencji.

Szerzej

Normalna zmienna wskazująca na obiekt jest silną referencją. Dopóki istnieje, GC nie zwolni obiektu. WeakReference jest użyteczne np. w cache, gdzie chcesz korzystać z obiektu, jeśli jeszcze istnieje, ale nie chcesz blokować jego usunięcia przez GC.

var weak = new WeakReference<User>(user);
if (weak.TryGetTarget(out var cachedUser))
{
    // obiekt jeszcze żyje
}
Dispose vs Finalizer
Krótko

Dispose() zwalnia zasoby deterministycznie, a finalizer jest awaryjnym mechanizmem uruchamianym przez GC w nieokreślonym czasie.

Szerzej

IDisposable stosuje się, gdy klasa trzyma zasoby wymagające jawnego zwolnienia. Finalizer, czyli destruktor w C#, powinien być rzadki i używany głównie przy bezpośrednim trzymaniu zasobów niezarządzanych. Finalizery obciążają GC i nie gwarantują natychmiastowego zwolnienia zasobów.

Memory leak on production scenario
Krótko

W .NET memory leak najczęściej oznacza, że aplikacja nadal trzyma referencje do obiektów, których już logicznie nie potrzebuje.

Szerzej

Typowe przyczyny to statyczne kolekcje, nieodpinane eventy, źle zaprojektowany cache, singleton trzymający scoped zależności, niezwolnione zasoby, rosnące kolejki albo błędne użycie HttpClient i streamów. Na produkcji sprawdza się metryki pamięci, dump procesu, liczbę obiektów i ścieżki referencji w narzędziach typu dotMemory, PerfView, Visual Studio Diagnostics albo dotnet-dump.

Concurrency & threading

What is a semaphore?
Krótko

Semaphore ogranicza liczbę wątków lub operacji, które mogą jednocześnie wejść do danego fragmentu kodu.

Szerzej

W .NET często używa się SemaphoreSlim, szczególnie w kodzie async. Przykład: chcesz pobierać dane dla 1000 elementów, ale maksymalnie 5 requestów naraz. Semaphore działa jak licznik dostępnych wejść: WaitAsync() zajmuje miejsce, a Release() je oddaje.

private readonly SemaphoreSlim _semaphore = new(5);

await _semaphore.WaitAsync();
try
{
    await DoWorkAsync();
}
finally
{
    _semaphore.Release();
}

.NET platform & build

.NET Framework vs .NET Core / modern .NET
Krótko

.NET Framework to starsza, głównie Windowsowa platforma. .NET Core i obecny .NET są cross-platform, szybsze i aktywnie rozwijane.

Szerzej

.NET Framework jest używany głównie w starszych aplikacjach Windows, WebForms, WCF i legacy ASP.NET. .NET Core był nową, modularną platformą działającą na Windows, Linux i macOS. Od .NET 5 nazwa została uproszczona do ".NET". Nowe projekty zwykle tworzy się na aktualnym .NET, np. .NET 8 lub .NET 9.

JIT vs AOT in .NET
Krótko

JIT kompiluje kod w czasie działania aplikacji, a AOT kompiluje go wcześniej, przed uruchomieniem.

Szerzej

W typowym .NET kod C# jest kompilowany do IL, a potem JIT zamienia IL na kod maszynowy podczas działania programu. AOT może skrócić startup i zmniejszyć zależność od runtime, ale może mieć ograniczenia, np. przy reflection lub dynamicznym generowaniu kodu. Native AOT jest przydatne m.in. w małych usługach, CLI i serverless.

Debug vs Release build configuration
Krótko

Debug jest do developmentu i debugowania, Release jest zoptymalizowany pod uruchomienie produkcyjne.

Szerzej

Debug zwykle zawiera pełniejsze symbole debugowania, mniejsze optymalizacje i ułatwia krokowe przechodzenie przez kod. Release włącza optymalizacje kompilatora, może inline'ować metody i usuwać nieużywany kod. Dlatego bugi czasem ujawniają się tylko w Release, szczególnie przy współbieżności lub zależności od timingów.

What is code obfuscation?
Krótko

Obfuskacja utrudnia odczytanie i zrozumienie skompilowanego kodu po dekompilacji.

Szerzej

W .NET kod IL można relatywnie łatwo dekompilować, dlatego obfuskatory zmieniają nazwy klas i metod, zaciemniają flow programu, szyfrują stringi albo utrudniają reverse engineering. Nie daje to pełnego bezpieczeństwa, ale zwiększa koszt analizy kodu. Nie należy traktować obfuskacji jako zamiennika prawdziwych mechanizmów bezpieczeństwa.

Design, DI & architecture

DI - types of object lifetimes
Krótko

W ASP.NET Core DI są trzy główne lifetime'y: Transient, Scoped i Singleton.

Szerzej

Transient tworzy nową instancję przy każdym żądaniu zależności. Scoped tworzy jedną instancję na scope, w web API zwykle na jeden request HTTP. Singleton tworzy jedną instancję na całe życie aplikacji. Trzeba uważać, żeby singleton nie trzymał zależności scoped, np. DbContext.

services.AddTransient<IEmailSender, EmailSender>();
services.AddScoped<IOrderService, OrderService>();
services.AddSingleton<IClock, SystemClock>();
ICloneable interface in Prototype pattern
Krótko

ICloneable pozwala tworzyć kopię obiektu, ale w praktyce bywa problematyczne, bo nie mówi jasno, czy kopia jest płytka czy głęboka.

Szerzej

Prototype polega na tworzeniu nowych obiektów przez klonowanie istniejącego wzorca. ICloneable.Clone() zwraca object i nie definiuje semantyki kopii. Dlatego często lepiej zrobić własną metodę, np. Copy(), DeepCopy() albo konstruktor kopiujący.

public Order Copy()
{
    return new Order(Id, CustomerName, Items.ToList());
}
Order Service Code Review
Krótko

W code review serwisu zamówień patrzysz na poprawność logiki biznesowej, transakcje, walidację, obsługę błędów, testowalność i zależności.

Szerzej

Dobry OrderService nie powinien być "God service". Warto sprawdzić, czy logika domenowa nie jest ukryta w kontrolerze, czy operacje zapisu są transakcyjne, czy nie ma duplikacji, czy zależności są przez interfejsy, czy nie ma mieszania odpowiedzialności, np. płatności, maili, magazynu i faktur w jednej metodzie. Ważne są też testy przypadków brzegowych: brak produktu, brak stanu magazynowego, nieudana płatność, retry, idempotencja.

Tematy widoczne częściowo na zdjęciu

What type would you use to store date/time with timezone in C#?
Krótko

Najczęściej użyłbym DateTimeOffset, bo przechowuje datę i czas razem z offsetem względem UTC.

Szerzej

DateTime może być lokalny, UTC albo nieokreślony, co łatwo prowadzi do błędów. DateTimeOffset jest lepszym wyborem dla momentów w czasie, które mają informację o przesunięciu. Jeśli potrzebujesz pełnej strefy typu Europe/Warsaw i zasad DST, przechowuj dodatkowo identyfikator strefy albo użyj biblioteki typu NodaTime.

Which types are delivered with .NET?
Krótko

.NET dostarcza bogaty zestaw typów bazowych, np. string, int, bool, DateTime, kolekcje, wyjątki, delegaty i typy asynchroniczne.

Szerzej

Podstawą jest Base Class Library. Zawiera typy prymitywne, kolekcje jak List<T> i Dictionary<TKey,TValue>, typy IO, networking, reflection, threading, LINQ, taski oraz wiele API frameworkowych. Na rozmowie warto podkreślić różnicę między aliasami C# jak int a typami CLR jak System.Int32.

Uwaga: dwa tytuły w źródłowym materiale były ucięte, więc zostały odtworzone jako najbardziej prawdopodobne tematy: date/time type oraz built-in .NET types.
